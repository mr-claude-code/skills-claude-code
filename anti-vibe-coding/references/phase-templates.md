## Template: docs/specs/[domain]/spec.md (Phase 3 - Specifications)

This artifact follows the SDD format: formal requirements with RFC 2119 keywords and Given/When/Then scenarios. Each domain gets its own spec file. Specs describe **observable behavior**, not implementation details.

```markdown
# [Domain Name] Specification

> Generated by Anti-Vibe Coding methodology (Spec-Driven Development)
> Date: [date]

## Purpose

[Brief description of what this domain covers and why it exists]

## Roles and Permissions

| Role | Description | Permissions |
|------|-------------|-------------|
| [role] | [who they are] | [what they can do] |

---

## Requirement: [Requirement Name]

[The system SHALL/SHOULD/MAY [observable behavior description]]

### Scenario: [Descriptive scenario name — happy path]
- **Given** [precondition or context]
- **When** [action or trigger]
- **Then** [expected outcome]

### Scenario: [Descriptive scenario name — error case]
- **Given** [precondition or context]
- **When** [action or trigger]
- **Then** [expected error behavior]

### Scenario: [Descriptive scenario name — edge case]
- **Given** [unusual but possible precondition]
- **When** [action or trigger]
- **Then** [expected behavior]

---

## Requirement: [Next Requirement Name]

[The system SHALL/SHOULD/MAY [observable behavior description]]

### Scenario: [Scenario name]
- **Given** [precondition]
- **When** [action]
- **Then** [outcome]

---

## Validations

| Field | Rule | Error Message |
|-------|------|---------------|
| [field] | [validation] | [message] |

## Global Rules

[Rules that apply across all requirements in this domain]
- [e.g., "All actions SHALL be logged with timestamp and user_id"]
- [e.g., "Soft delete on all entities"]
- [e.g., "Rate limiting of 100 requests/minute per IP"]
```

**Key guidelines for writing good specs:**
- Focus on **observable behavior**, inputs/outputs, and testable conditions
- Avoid implementation details, library choices, and internal naming conventions
- Use `SHALL` for mandatory, `SHOULD` for recommended, `MAY` for optional
- Every requirement should have at least one happy path and one error scenario
- Scenarios should map directly to test cases in Phase 4

**Delta Specs (for brownfield projects):**

When adding features to an existing codebase, use delta format to show only what changed:

```markdown
## ADDED Requirements
### Requirement: [New Feature]
The system SHALL [new behavior]
#### Scenario: [scenario name]
- Given [precondition]
- When [action]
- Then [outcome]

## MODIFIED Requirements
### Requirement: [Existing Feature]
The system SHALL [updated behavior]
(Previously: [old behavior])

## REMOVED Requirements
### Requirement: [Deprecated Feature]
Reason: [why it's being removed]
```
